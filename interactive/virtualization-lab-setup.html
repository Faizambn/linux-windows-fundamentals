01 the magic underneath

What virtualization is

A hypervisor is software that carves one physical machine into several virtual machines (VMs) — each thinking it's a real computer with its own OS. There are two types, and the difference is where the hypervisor sits. Toggle between them.

which one you'll useAs a learner you'll use a Type 2 hypervisor — VirtualBox (free) or VMware Workstation — because it installs like any app on top of your existing Windows/Mac. Type 1 (VMware ESXi, Microsoft Hyper-V, Proxmox) runs directly on server hardware and powers data centres and the cloud.

// your laptop's OS = the "host"; each VM inside it = a "guest".

02 a computer inside a file

Anatomy of a VM

A VM is just virtual hardware: slices of your real CPU, RAM, and disk, presented to a guest OS as if they were physical. The whole machine is really a few files on your disk — which is what makes VMs so easy to copy, move, and reset. Click each part.

🖥️ Virtual Machine
  • Runs a full guest OS (its own kernel)
  • Strong isolation from the host
  • Heavy — GBs of RAM/disk each
  • Best for: whole systems, malware analysis
📦 Container (e.g. Docker)
  • Shares the host's kernel
  • Lighter isolation, starts in seconds
  • Tiny — ships just the app + its deps
  • Best for: running apps/services fast
the isolation caveatVMs are isolated, but not perfectly — VM escape (malware breaking out of the guest onto the host) is a real, if rare, attack class. Containers share the host kernel, so they're weaker boundaries than VMs. For detonating real malware, a VM with isolated networking + snapshots is the safe choice.

// a VM's disk is a file (.vdi / .vmdk). Copy it = clone the whole machine.

03 the part everyone gets wrong

VM network modes

How a VM connects to the world is the single most important — and most confused — lab setting. Pick the wrong mode and either your lab won't work, or (worse) malware you're studying escapes onto your real network. Choose a mode and see exactly what the VM can reach.

the golden rule of malware analysisWhen studying malware, use Internal or Host-only networking so it's sealed off from the internet and your real LAN — never Bridged, which puts the infected VM directly on your home network where it can spread. Flip to NAT only briefly for updates, then seal it back up.

// in VirtualBox this is Settings → Network → "Attached to:" — the dropdown that decides everything.

04 your training ground

Build a lab + snapshots

A classic starter lab: an attacker VM (Kali Linux) and one or more victim VMs (a Windows box, or a deliberately-vulnerable machine like Metasploitable), all on an isolated internal network so they can reach each other but nothing else.

Snapshots — the time machine
🟢 VM state: clean & healthy
Take a snapshot before you do anything risky.
why snapshots are a superpowerA snapshot saves the VM's exact state. Take one, detonate malware to study it, observe the damage — then revert and the VM is instantly pristine again, as if nothing happened. This is what makes safe, repeatable malware analysis possible. No cloud, no internet needed — perfect for a local lab.
