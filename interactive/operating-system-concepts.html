An operating system's real job is drawing lines — between programs and hardware, between one app and another, between trusted and untrusted code. Almost all of system security is about defending those lines. Let's see them.

01 the great divide

Kernel vs User Space

The CPU can run code at different privilege levels. The OS uses two: kernel mode (ring 0) can touch any hardware and any memory; user mode (ring 3) is fenced in — your apps live here and can't touch hardware directly. To do anything privileged, a program must ask the kernel through a guarded doorway. That fence is the most important security boundary on the machine.

attack & defenseAttackers arrive in user space and try to climb into the kernel — via a kernel/driver bug — because kernel code sits below the security tools and can hide from them (a rootkit). Defenders use least privilege, patching, driver signing, and EDR (a sensor that watches the boundary from inside the kernel) to spot the crossing.

02 what "running" means

Processes & Threads

A process is a running program with its own private memory. Inside it, threads are separate lines of execution that share that memory — which is how a program does several things at once. The OS constantly moves each process through a lifecycle of states as it shares the CPU.

attack & defenseBecause threads share the process's memory, malware uses process injection — sneaking a malicious thread into a trusted process (like a browser) so its bad activity wears a trusted name. Defenders (via EDR) watch for one process creating a thread inside another, a classic injection tell.

03 everyone gets their own world

Virtual Memory & Paging

Every process is given the illusion that it owns all of memory. In reality the OS hands out virtual pages and secretly maps them, through a page table, to real physical frames in RAM — or, if RAM is full, to a slower spot on disk. Click a virtual page to follow the mapping.

Click a virtual page on the left to see where it really lives in RAM.
attack & defenseThis mapping is what keeps one process from reading another's memory — a core isolation boundary. Exploits try to break it (read/write memory they shouldn't); defenses like ASLR randomise where things land in memory so an attacker can't predict addresses, and the hardware blocks cross-process reads.

// a "page fault" is just the CPU saying "that page isn't in RAM yet" — normal, but a flood of them (thrashing) means memory is exhausted.

04 asking for power

System Calls & Drivers

When your program wants to open a file or send a packet, it can't do it directly — it makes a system call, the guarded request into the kernel. The kernel then talks to a device driver, the specialised code that knows how to operate a specific piece of hardware. Drivers run in kernel mode. Press play to follow one request down and back.

attack & defenseDrivers are a prized target: because they run in the kernel, a buggy one is a direct path to full control. In a BYOVD ("bring your own vulnerable driver") attack, criminals load a legitimately-signed but flawed driver and abuse it to run kernel code. Defenders block known-vulnerable drivers and monitor system-call patterns for abuse.
