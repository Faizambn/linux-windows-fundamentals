01 the lay of the land

The memory hierarchy

A computer doesn't have one kind of memory — it has a stack of them, trading size against speed. Tiny and blazing fast at the top (inside the CPU), enormous and slow at the bottom. The bar width below shows capacity (how much fits); the colour shows speed. Click any layer to go inside it.

◄ faster · smallerbigger · slower ►
Click a layer to inspect it — its real capacity, latency, and what actually lives there.
why a security person caresCredentials and encryption keys live in the fast top layers (registers, cache, RAM) while in use — never touching the disk. That's the entire premise of memory forensics: the secrets are up here, and they vanish on power-off.

// see your own machine's layers: Linux → lscpu (look for L1d/L1i/L2/L3) · Windows → Task Manager → Performance → CPU

why this mattersThis is the insight behind fast software: a cache miss that falls back to RAM is like a 3-second task suddenly taking 6 minutes. Reaching disk is like waiting days. Performance engineering — and a lot of exploitation and defense timing — lives in these gaps.

03 follow the data

The journey of a single byte

You open a file. That byte starts life on the disk and has to climb all the way up to a CPU register before it can be used. Watch it travel — and watch the clock, because most of the total time is spent on the very first hop.

why a security person caresThis climb is why caching exists — and why a program is copied from disk into RAM to run, becoming a live process. "Fileless" malware never lands on disk at all; it lives only up in RAM to stay invisible to scanners that watch storage.

// the takeaway: reaching disk is ~100,000× slower than reaching cache. The whole memory hierarchy exists to avoid that trip whenever possible.

04 the engine room

Inside the CPU core: fetch · decode · execute

Down at the core, the CPU repeats three steps billions of times a second: fetch the next instruction (the Program Counter says which), decode what it means, then execute it in the ALU and write the result to a register. Run this real 5-instruction program and watch it compute 5 + 3.

why a security person caresThe Program Counter decides what runs next. If an attacker can overwrite it (e.g. via a buffer overflow), they redirect the CPU into their code. "Control the PC = control the machine" is the foundation of memory-corruption exploitation — and of the defenses built to stop it.

05 the fine print

Word size & byte order

32 vs 64-bit is how large an address the CPU handles at once — which caps how much RAM it can even see. 32 bits hits a wall near 4 GB; 64 bits reaches a number so vast the limit stops mattering. That ceiling is why the whole world moved to 64-bit.

~16 exabytes
addressable memory — effectively limitless for now

Endianness is the order the bytes get stored in. Take the value 0x12345678. Big-endian stores it the way we read it; little-endian — what your Intel/AMD laptop uses — stores the small end first, so it looks reversed in a hex dump.

why a security person caresGrab the wrong build (x86 vs x64) and tools fail silently. And the first time a value looks "backwards" in Wireshark or a hex editor, endianness is the reason — not a mistake in your reading.

// check yours: Windows → Settings → System → About → "System type" · Linux → uname -m (x86_64 = 64-bit)
